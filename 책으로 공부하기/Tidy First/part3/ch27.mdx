# 옵션과 현금흐름 비교

"코드 정리가 먼저인가?" 라는 흥미로운 질문을 만드는 경제적 줄다리기를 하고 있음.

- 현금 흐름 할인 - 높은 확률로 먼저 돈을 벌고, 낮은 확률로 나중에 돈을 쓰기 (코드 정리는 나중에)
- 옵션: 나중에 더 많은 돈을 벌기 위해서 설사 정확한 방법을 모르더라도 지금 돈을 쓰기 (옵션이 생길일이 명백하다면 코드 정리를 선행하고, 동작 변경 후에도 정리할 내용이 있다면 하기)

#### 코드 정리를 선행해야 할 경우,

- 비용 (코드 정리) + 비용(코드 정리 후 동작 변경) < 비용(바로 동작 변경)

단, 코드 정리의 경계를 잘 설정하기

#### 고민을 하게 되는 경우

- 비용 (코드 정리) + 비용(코드 정리 후 동작 변경) > 비용(바로 동작 변경)

단기적인 경제성 때문에 코드 정리가 망설여지더라도 코드 정리를 먼저하고 싶은 경우도 있으며, 이 코드 정리 과정에서 동작 변경까지 하게 되는 경우도 존재.

이 지점은 판단의 갈림길이다.

코드 정리 규모에서 코드 정리의 경제성을 정확하게 계산할 수 없고, 계산하려는 시도 자체가 무의미하다.  
대신, 2가지 중요한 형태의 판단력을 길러서 나중에 더 큰 일을 실행하려고 해야함

1. 소프트웨어 설계의 시기와 범위에 영향을 미치는 인센티브를 인식하는데 익숙해지기.
2. 대인 관계 기술을 자기 자신에게 연습하고, 이를 나중에 밀접하고 일하는 동료로부터 더 넓은 동료에게까지 활용하기

이 연습을 한다면, 제품의 생존과 번영이 걸려 있는 중요한 순간에 이를 때 언제 어떻게 설계해야 하고 언제 설계하지 말아야 하는지 직감적으로 알 수 있게 된다.
