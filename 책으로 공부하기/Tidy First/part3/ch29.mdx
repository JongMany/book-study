# 결합도

비싼 프로그램 ➡️ 한 요소를 변경하려면 다른 요소도 변경해야 한다.
저렴한 프로그램 ➡️ 좁은 범위의 코드 변경만 필요해야 한다.

이러한 변경 감염 특성은 '결합도'임.

- 한 요소를 변경하면 다른 요소도 함께 변경해야 하는 경우, 두 요소는 특정 변경과 관련하여 서로 결합되어 있는 것

```
caller()
  called()

called() // 이 이름을 변경하면 호출하는 코드도 변경해야 함
  ... // 함수 본문은 형식을 변경해도 호출하는 코드를 변경할 필요가 없음.
```

결합도가 의미를 지니려면 어떤 변경 사항과 관련하여 결합되어 있는지 말해야 한다.

'결합도 분석'은 단순히 프로그램을 소스 코드를 보는 것만으로는 부족함. 두 요소가 결합되어 있는지 여부를 판단하려면 먼저 어떤 변경이 발생했거나 발생할 가능성이 있는지 알아야 함.

결합도는 두 요소만의 문제가 아님. 결합도가 가진 2가지 성질 때문에 결합도를 주목하는 것임

1. 일대다(1-N): 어떤 변경이 일어나면 한 요소는 여러 요소와 결합이 일어남
2. 연쇄작용: 일단 변경이 일어나면 한 요소에서 다른 요소로 변경이 파급되고, 그 변경은 그 자체로 또 다른 변경을 촉발하고, 스스로도 변경을 촉발할 수 있음.

일대다 문제는 개발도구를 통해 어느정도 해결할 수 있음 (IDE의 리팩터링 기능)

연쇄적인 변경은 소프트웨어 설계를 통해 연쇄적인 변경의 확률과 규모를 줄일 수 있게 됨.

** 한 서비스를 변경하면 다른 서비스도 변경해야 한다는 것을 알아야 함**
