# 얽힘 풀기

코드의 동작 변경을 하고 있다고 가정하자.  
이때 변경을 쉽게할 코드 정리를 알게 되어 코드 정리를 진행하게 됨.  
그런 다음 테스트 케이스를 작성하고 동작을 변경. 그랬더니 코드 정리할 곳이 또 생김...

이런 과정을 반복하니 다음과 같은 결과를 얻게 됨

1. 변경 대상인 동작을 모두 알게 됨
2. 그 동작을 쉽게 변경하려면 어떤 코드를 정리해야 하는지 알게 됨.
3. 문제는 정리한 코드와 변경할 동작이 함께 얽혀 버림

우리는 3가지 선택지가 있음

1. 그대로 배포
2. 코드 정리와 변경 사항을 별도로 하나 이상의 PR로 나누거나 하나의 PR을 여러 번의 커밋으로 나누기
3. 진행 중인 작업을 버리고 코드 정리를 선행하는 순서로 다시 시작
   (작업은 늘어나게 되지만 커밋과의 일관성은 분명해짐)

이 중 어느 것을 선택하든 어느 정도의 비용이 발생하지만, 3번째를 추천함  
다시 구현하면서 새로운 것을 발견할 가능성이 높기 때문에
