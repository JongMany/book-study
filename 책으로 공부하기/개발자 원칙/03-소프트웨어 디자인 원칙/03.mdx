프로그램을 작성하는 행위인 코딩 자체에 대한 원칙 (작은 단위의 코드에 적용되는 원칙)은 많음
KISS (Keep I Simple, Stupid): 코드를 작성할 때, 가장 간단한 방법으로 작성하라
DRY(Don't Repeat Yourself): 반복되는 기능이나 객체를 하나로 만들어서 관리하라
YAGNI(You Ain't Gonna Need It): 코드를 적어나가다보면 갑자기 머리 속에서 떠오르는 필요한 기능이 있어서 막 적지만 결국 안 쓴다는 법칙

진짜 디자인 악취를 제거하려면 위의 디자인 원칙을 따를 것이 아니라(인과 관계가 없으므로) 진짜 원인을 찾아서 제거해야 함

SW 제품을 만들 때 쯤에 발생하는 다음과 같은 생각을 가지는 경우가 많음

- SW는 제대로 디자인하고 만들기에는 늘 사람과 시간, 돈이 부족함
- SW는 한사람이나 팀이 처음부터 끝까지 개발하지 않고 언제가는 히스토리를 모르는 사람이 개발을 맡게 됨
- SW는 한 번 만들어지고 나면 아주 오랫동안 사용됨
- 이미 릴리즈된 SW는 다음 버전이나 개선 때 더 빨리 만들 수 있다고 믿음
- SW 세계에서 변하지 않는 것은 '시간/사람/돈에 대한 결핍' 외에는 아무것도 없음
- SW에 대한 요구사항은 의도적이든 아니면 자연적이든 시간이 지남에 따라 늘 변함

### 디자인이란?

디자인 = 설계 = '목적'에 따라 '실제적인 계획'을 세우고, '구체적으로 도면'을 그려 명시하는 것

실제적인 계획

- 간트 차트? WBS?
- Waterfall? Agile?

구체적인 도면

- 설계 / 제작 도구 => 그러나 아직 표준화된 설계 제작 도구는 없음
- 프로그램 단위의 설계 => DRY ? KISS?
- 큰 객체의 설계 => SOLID?

### 설계와 요구사항

설계 - 제품이 요구사항을 만족시키는 것을 증명하는 조건을 정의하는 행위
설계 전에 요구사항을 잘 정의해야함

소비자가 자신의 요구사항을 모른다면 제품을 만드는 생산자라도 어떤 제품을 만들지 명확하게 정의해야 함
요구사항이 정해지면 설계 단계에서는 만들 제품이 주어진 요구사항을 잘 충족하는지 증명할 수 있는 조건을 정의해야 함  
-> 이 과정에서 요구사항에 나타나지 않았던 수치가 정해짐 (단어적인 설계나 개념이 숫자적 개념으로 변경됨)

설계 조건이 정해지면 테스트에 필요한 조건도 정확하게 정해짐 -> 테스트 계획 / 조건이 부산물로 나옴  
즉, 설계에는 TDD, BDD, DDD 개념이 들어가 있음

> 요구사항 예시) 드론이 화성에서 비행할 수 있어야 한다.  
> 문제 사항: 화성과 지구는 5천만 KM 떨어져 있음  
> 설계의 관점을 변경해야 함 -> 요구사항을 만족한다는 것을 증명할 수 있는 조건을 만들어야 함

### 소프트웨어 설계 원칙: 통합적으로 설계하라

SW 설계시 최대한 많은 정보를 기반으로 요구사항을 정리해야 함
기획: 제품과 관련된 전략적 목표(재무적 목표, 시장 점유율 목표, MAU, 사용자 만족도 등)을 기반으로 기획서에 상세히 요구사항을 기술해야 함
-> SRS(소프트웨어 요구사항 명세)

SRS를 설계한 후에는 종합적으로 설계를 해야함

- 요구사항을 바탕으로 제품이 가져야 하는 특성을 종합적이고 전체적으로 설계해야함
- 종합적으로 설계하는 방법: 명시적 설계 / 내재적 설계  
  둘의 차이 산출물이 SRS나 기본 요구사항에 직접 연결되는지
