# 유비쿼터스 언어

도메인 전문가와의 의사소통을 위해서는

- 모델을 만들 때,
  - 모델에 대한 아이디어
  - 포함되어야 할 요소
  - 요소들을 연결할 방법 (어떤 것들이 관계가 있고, 어떤 것들이 관계가 없는지)
  같은 정보를 교환해야 함
- 팀 멤버끼리 도메인에 관해 토의할 수 있는 공통 언어를 갖지 못하면 위험함
  - 도메인 전문가는 자신들의 전문 용어를 사용
  - 기술 팀 멤버는 설계의 관점 만을 위해 팀 멤버만의 언어를 사용

---

도메인 주도 설계의 핵심 원칙

- 모델 기반의 언어를 사용하는 것
- 모델
  - 소프트웨어와 도메인이 서로 교차하는 지점
- 모델 기반 언어를 사용하는 것이 적절함
  - 모델 기반 언어 = 유비쿼터스 언어
- 코드 또한 모델에 부합하도록 설계하기
- 역할
  - 도메인 전문가
    - 어색하거나 오해를 일으키는 용어, 구조에 반대해야 함
  - 개발자
    - 설계에서 나타날 수 있는 모호함이나 불일치를 늘 검토해야 함

모델링 도구

- UML
  - 간단한 상황에서는 좋음 but SW가 비대해지면 위험함
  - 개념들의 의미, 객체들은 무슨일을 하는지에 대해서는 표현하기 어려움
  → 모든 것을 포함하는 다이어그램을 작성하는 것은 불가능함
- 문서
  - 문서에 간단한 UML을 추가하여 의사소통을 하자
  - 긴 문서 작성에 유의
    - 작성하는 시간도 오래 걸리고, 완성되기 전에 폐기되어야 할 ㅜ 있음
  - 문서 = 모델과 동일한 내용이어야 함
